# Wynne's ZOO

这是一个以“动物园”为包装主题的前端游戏原型项目。  
当前代码已经完成了第一个核心模块: `资源获取层玩法`，也就是以开箱/翻格子/连团结算为核心的 Slot-like 体验。  
你接下来准备开始的第二个模块，是 `动物园模拟经营`，其中最重要的第一步就是让玩家可以建造动物栖息地。

README 已按当前仓库里的实际代码结构同步，不再只描述产品概念。

---

## 当前项目阶段

项目现在可以理解成双层结构:

- `模块 1: 资源获取层`
  - 已实现
  - 玩家在 4x4 棋盘中逐格开箱
  - 通过符号、连团、Bonus、Free Spin、猴子转盘、叠塔进度获得奖励
- `模块 2: 动物园经营层`
  - 还未正式开始落代码
  - 目标是承接模块 1 产出的资源
  - 玩家用资源解锁地块、建造栖息地、安置动物、升级园区

这意味着当前仓库已经有一个相对完整的“前台爽感循环”，而你接下来要做的是把这些奖励真正消费到“后端成长目标”里。

## 当前玩法逻辑概览

### 1. 主棋盘

- 主棋盘固定为 `4x4 = 16` 格
- 每一轮生成一套 `boardPlan`
- 玩家点击格子后翻开箱子，显示对应符号
- 当所有格子开完后，本轮结算结束并展示总结弹窗

### 2. 符号体系

当前代码里已经有完整的符号类型:

- `普通符号 S1-S5`
- `Wild`
- `Sticky Wild`
- `Bonus`
- `Monkey`

其中:

- `Wild` 参与连团匹配
- `Sticky Wild` 会跨轮停留指定回合数
- `Bonus` 会参与 Bonus 触发条件
- `Monkey` 会触发猴子动画和奖励转盘

### 3. 棋盘生成方式

项目不是简单地“按权重随机每一格”，而是做了更接近产品控制的生成方式:

- 使用 `BLUEPRINTS` 定义每种棋盘蓝图
- 某些蓝图绑定 `template mask`
- 支持旋转/翻转模板变换
- 支持 `Sticky Wild` 跨轮继承后的专用权重池
- 支持 `Bonus` 蓝图的连续未出保底

也就是说，当前的棋盘生成已经具备明显的“策划可控性”，而不是纯随机。

### 4. 实时连团结算

玩家每翻开一个格子，都会立即触发一次实时检测:

- 找当前棋盘上每种符号的最佳连团
- 达到最小连团数后立刻结算
- 奖励按 `clusterPayout` 配置计算
- 连团奖励会转化为叠塔积木、分数和表现动画

这个版本的结算更接近“Cluster 连团奖励”，不是传统固定赔线老虎机。

### 5. Bonus / Free Spin / Monkey 奖励链路

当前代码里已经把几个主要强反馈玩法串起来了:

- `Bonus Trigger`
  - 由 `Bonus` 与 `Wild` 的触发条件驱动
  - 实际 Bonus 玩法逻辑在 `js/bonus.js`
  - 该模块由 `script.js` 按需懒加载
- `Free Spin`
  - 进入单独的免费回合状态
  - 支持特殊尺寸大箱子 `2x2 / 3x3 / 4x4`
  - 有单独 HUD、结果结算和表现层
- `Monkey Wheel`
  - 开到猴子符号后，先播猴子动画，再进入奖励转盘
  - 转盘奖励可加叠塔、补 Bonus、拉满进度等

### 6. 叠塔与倍率成长

上半区的塔体不是纯装饰，而是当前版本的成长反馈主轴之一:

- 连团奖励会飞入塔体
- 塔体高度驱动倍率进度
- 倍率影响本轮奖励表现
- 塔体还承担视觉成长与反馈强化作用

### 各文件职责

`index.html`

- 页面骨架
- 主棋盘、塔体、Bonus Overlay、Free Spin Overlay、结算弹窗都定义在这里
- 默认只直接加载:
  - `config/game-config.js`
  - `script.js`

`script.js`

- 当前项目的主运行时入口
- 做了默认配置和外部配置合并
- 维护主棋盘状态、Free Spin 状态、Sticky Wild 状态
- 负责棋盘生成、开箱、实时结算、猴子转盘、Bonus/Free Spin 触发
- 会按需懒加载:
  - `js/effects.js`
  - `js/bonus.js`

`config/game-config.js`

- 对外部数值和 UI 参数做集中覆盖
- 当前很多调参都不需要改 `script.js`

`js/bonus.js`

- Bonus 子玩法模块
- 通过 `window.BonusGame` 暴露接口
- 包括 Bonus 棋盘、宝箱、触发条、奖励结算、Bonus 结果界面

`js/effects.js`

- 粒子、浮字、冲击波、礼花等纯表现层 API
- 通过 `window.GameEffects` 暴露给主逻辑使用

`style.css` / `effects.css`

- 主界面布局、棋盘、奖励界面、Free Spin、Bonus、转盘和粒子样式

## 运行时架构

### 配置层

项目运行时依赖一个全局配置入口:

- `window.APP_CONFIG`

当前主要分成两部分:

- `game`
  - 棋盘大小
  - 符号权重
  - 蓝图权重
  - 连团倍率
  - Bonus/Free Spin 参数
  - Sticky Wild 回合数
- `ui`
  - CSS 变量
  - 棋盘位置
  - 叠塔偏移
  - 箱子尺寸与偏移

这套结构对后续接入“动物园经营层”是有利的，因为你可以继续沿用同样的做法，把经营层参数外置到独立配置中。

### 状态层

当前主逻辑里至少有三套关键状态:

- `STATE`
  - 主棋盘运行时状态
  - 当前棋盘内容、已翻开格子、回合奖励、实时结算记录等
- `FREE_SPIN_STATE`
  - Free Spin 专用状态
  - 剩余次数、当前盘面、奖励、是否暂停等
- `STICKY_WILD_STATE`
  - Sticky Wild 跨轮继承状态
  - 记录哪些位置在后续回合强制出现

此外还有一个重要的持久化点:

- `localStorage['wynnesZoo.bonusDryStreak']`
  - 用于记录 Bonus 蓝图未出的连续次数
  - 达到阈值后触发保底权重池

### 模块加载方式

当前不是模块化打包项目，而是原生浏览器运行模式:

- 首屏只进 `script.js`
- 需要表现层时才加载 `js/effects.js`
- 需要 Bonus 子玩法时才加载 `js/bonus.js`

这个方式对于原型开发是足够轻的，适合继续快速迭代第二模块。

## 当前一局的真实流程

可以把现在代码里的主流程理解成:

1. `initGame()`
   - 重置主状态
   - 生成新棋盘蓝图
   - 重建 4x4 箱子 UI
2. 玩家点击格子
   - `handleCellClick()`
   - 播开箱动画
   - 获取该格子的符号数据
3. `applySafeReward()`
   - 显示符号
   - 处理 Sticky Wild 继承
   - 处理 Bonus / Wild 计数
   - 如果是 Monkey，进入猴子动画 + 转盘奖励
4. `settleRealtimeRewardsForCurrentBoard()`
   - 检查当前所有已揭示格子的最佳连团
   - 触发奖励、表现和叠塔增长
5. 触发额外玩法
   - `maybeTriggerBonusGame()`
   - `maybeTriggerFreeSpinGame()`
6. 所有箱子开完
   - 汇总本轮结果
   - 展示结算弹窗
   - 玩家重新开始下一局

这条链路已经能稳定构成一个完整的资源获取循环。

## 第二模块: 动物园模拟经营 / 栖息地建造

这是当前 README 里最需要提前定义清楚的部分。

### 目标

把当前模块 1 产出的奖励，转成动物园经营层的长期成长动力。

推荐的核心关系:

- 模块 1 负责 `产出`
- 模块 2 负责 `消费`、`成长`、`展示`

即:

`开箱奖励 -> 经营资源 -> 建造栖息地 -> 安置动物 -> 提升园区价值 -> 反哺前台玩法`

### 建议先做的最小可玩版本

如果你要开始第二模块，我建议先只做下面四件事:

1. `园区地块`
   - 有若干可解锁的建造格子
   - 玩家可以选中空地
2. `栖息地建造`
   - 每种动物对应一个基础栖息地类型
   - 支持建造、占地、解锁、升级
3. `动物入驻`
   - 已解锁动物可以放入合适的栖息地
   - 栖息地显示动物形象和基础属性
4. `经营产出`
   - 栖息地产出游客值、金币或门票收益
   - 经营层数值反过来影响模块 1 的某些掉落或倍率

只要这四步打通，整个项目就会从“纯爽感原型”升级成“有长期目标的产品雏形”。

### 推荐资源模型

建议不要一上来做过多资源种类，先控制在 4 类以内:

- `金币`
  - 通用建造和升级货币
- `建材`
  - 建造栖息地和园区设施
- `动物碎片`
  - 解锁具体动物
- `园区星级`
  - 作为全局成长门槛

这样最容易把当前已有玩法接进来:

- 主局结算产出金币
- Bonus / Free Spin 提供建材或碎片加成
- 稀有蓝图提高高品质奖励出现率
- 栖息地升级提升后续收益效率

### 推荐把“栖息地”当成第二模块的最小核心对象

先不要把“整个动物园系统”做得太大。  
先把 `Habitat` 这个对象做扎实，会比直接铺大地图更稳。

一个栖息地最少可以有这些字段:

```js
{
  id: 'habitat_grassland_001',
  type: 'grassland',
  animalType: 'zebra',
  level: 1,
  capacity: 2,
  comfort: 60,
  popularity: 20,
  buildCost: {
    coin: 500,
    material: 20
  },
  upgradeCost: {
    coin: 300,
    material: 10
  },
  unlocked: true,
  placed: true
}
```

### 建议的模块拆分

如果你马上开始写第二模块，建议不要继续把所有逻辑塞进 `script.js`。  
当前 `script.js` 已经很大，第二模块最好从一开始就拆开。

推荐目录:

```text
js/
├── bonus.js
├── effects.js
└── zoo/
    ├── zoo-state.js
    ├── habitat-system.js
    ├── zoo-ui.js
    └── zoo-balance.js
```

职责建议:

- `zoo-state.js`
  - 园区存档状态
  - 地块、栖息地、动物库存、资源库存
- `habitat-system.js`
  - 建造、升级、放置、解锁规则
- `zoo-ui.js`
  - 园区界面、建造面板、栖息地详情面板
- `zoo-balance.js`
  - 建造成本、升级成长、收益系数

### 与当前模块 1 的接入点

第二模块接入时，优先考虑下面几个位置:

- `Bonus` 结束后
  - 可以发放建材/动物碎片
- `Free Spin` 结束后
  - 可以发放稀有经营资源
- 主局结算弹窗前后
  - 把本轮奖励同步进园区资源池
- 蓝图或特殊符号命中时
  - 触发“动物线索 / 图纸 / 栖息地券”

最关键的是:

不要让经营层奖励完全独立于模块 1。  
要让玩家明确知道，“我在前台开出来的奖励，真的能拿去建设动物园”。

## 当前 README 给你的结论

从代码层面看，这个项目目前不是“动物园经营游戏未开始”，而是:

- `模块 1 已经相当完整`
- `模块 2 还没有正式落地`
- 两者之间已经有很明确的承接空间

因此，下一阶段最合理的开发顺序不是继续往 Slot 里堆玩法，而是:

1. 定义经营资源
2. 建立园区状态
3. 实现栖息地建造
4. 让模块 1 的奖励流入模块 2

这样项目会更像一个完整产品，而不只是单个原型玩法。
